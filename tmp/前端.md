基于 `FusionTrace` 的项目结构、工作流程以及您提出的四大功能需求，以下是一个专为该多源数据融合与溯源系统设计的**前端架构与功能设计方案**。

设计核心理念：**全景感知（Dashboard） + 深度溯源（Graph Investigation） + 战术透视（ATT&CK）**。

---

### 1. 总体布局与导航 (Layout)
*   **风格**：暗色系（Dark Mode），适合长时间分析，突出告警颜色的对比度。
*   **侧边导航栏**：
    1.  **态势总览 (Dashboard)**
    2.  **溯源画布 (Investigation)** —— *核心功能*
    3.  **战术分析 (ATT&CK Matrix)**
    4.  **资产与探针 (Sensor Fleet)** —— *对应5节点管理*
    5.  **威胁情报与归因 (Attribution)**

---

### 2. 详细功能模块设计

#### 2.1 态势总览 (Dashboard)
**目标**：满足【要求4】的靶场环境展示，以及【要求1】的数据采集状态概览。

*   **全网拓扑监控图 (Topology View)**：
    *   **元素**：动态展示 5 个节点的网络拓扑（分析中心、流量探针、3个受害主机 Agent）。
    *   **状态指示**：使用红/绿呼吸灯表示节点在线状态（Agent 心跳）。
    *   **流量连线**：节点间连线粗细代表流量大小，红色虚线连线代表检出的**隐蔽信道**（DNS/ICMP Tunnel，对应 `detection.zeek` 的产出）。
*   **实时数据流水 (Data Ingestion Pipeline)**：
    *   **仪表盘**：展示 EPS (Events Per Second)，区分“端侧日志（Auditd）”与“网侧日志（Zeek）”的入库速率。
    *   **时间对齐监控**：展示各节点 NTP 时间偏差值（满足【要求1】的时间序列对齐监控，若偏差>100ms 标红报警）。
*   **最新告警列表**：
    *   列出最近触发的 Wazuh 规则或 Zeek 异常检测，如“内存代码注入”、“WebShell 连接”。

#### 2.2 溯源画布 (Investigation Workbench) —— *最核心模块*
**目标**：满足【要求2】的实体关系图构建、【要求3】的攻击路径重建。
**技术支撑**：前端使用 `Cytoscape.js` 或 `G6` 对接后端 `Neo4j` 数据。

*   **交互式攻击图谱 (Interactive Graph)**：
    *   **节点设计**：
        *   `Process` (圆形, 蓝色): 显示 PID、进程名。若为“内存注入”进程（Auditd 检出），添加红色“Injected”角标。
        *   `File` (方形, 绿色): 显示文件名。敏感文件（如 `/etc/passwd`）高亮。
        *   `IP/Domain` (云形, 橙色): 显示 IP 地址。C2 服务器（情报命中）显示骷髅图标。
    *   **边（关系）设计**：
        *   带有方向的箭头，如 `SPAWNED` (父子进程)、`CONNECTED_TO` (网络连接)、`WROTE_TO` (文件篡改)。
        *   **时序回放轴**：底部提供时间滑块（Timeline Slider），拖动滑块，图谱动态展示攻击发生的先后顺序（满足【要求2】的时间线关联）。

*   **攻击路径高亮 (Path Reconstruction)**：
    *   **功能按钮**：“一键溯源”。
    *   **效果**：系统自动计算最短路径，将从 **初始入侵点** (如 WebShell 进程) -> **提权操作** (如 SUID 提权) -> **横向移动** -> **数据外传** (连接 C2) 的路径加粗并高亮显示（满足【要求3】的完整路径分析）。

*   **侧边详情面板 (Detail Panel)**：
    *   点击图谱中的节点，侧边栏弹出详细信息。
    *   **进程节点**：展示完整的命令行参数、父进程 ID、启动时间。
    *   **网络节点**：展示五元组、Payload 摘要、协议类型。
    *   **登录会话**：展示重建后的 User Session ID、源 IP、登录时长（满足【要求1】的会话重建）。

#### 2.3 战术透视 (ATT&CK Mapper)
**目标**：满足【要求2】的 ATT&CK 映射与【要求3】的 APT 组织特征匹配。

*   **动态 ATT&CK 矩阵热力图**：
    *   **展示**：复刻 MITRE ATT&CK 矩阵表格。
    *   **映射**：根据后端分析结果，点亮对应的技战术单元格（如 `T1059.004` Unix Shell）。
    *   **钻取**：点击高亮单元格，跳转到“溯源画布”中对应的具体的日志事件或图节点。

*   **APT 组织画像卡片 (Attribution Card)**：
    *   **匹配度仪表盘**：显示当前攻击行为与已知 APT 组织（如 APT28, Lazarus）的相似度百分比（基于 TTPs 匹配）。
    *   **指纹证据链**：列出支撑归因的证据，例如：“使用了特定的 Rootkit 特征”、“连接了已知的恶意 C2 域名”、“符合某组织的典型攻击剧本”。

#### 2.4 资产与探针管理 (Sensor Fleet)
**目标**：管理 5 节点靶场环境，确保【要求4】的验证环境可视。

*   **主机列表 (Agents)**：
    *   显示 Node 3, 4, 5 的 OS 版本、IP、内核版本。
    *   开关控制：远程下发开启/关闭 `auditd` 规则。
*   **流量探针配置 (Network Sensors)**：
    *   Node 2 (Zeek) 的状态。
    *   配置开关：开启/关闭特定的 Zeek 脚本（如开启 `detection.zeek` 用于检测 DNS 隧道）。

### 3. 特殊交互场景设计（针对具体要求）

#### 场景 A：隐蔽信道检测与分析
*   **界面触发**：Dashboard 收到“DNS Tunnel Detected” 告警。
*   **跳转**：进入流量分析详情页。
*   **可视化**：
    *   展示该 Session 的 **字节分布图**（Payload 熵值分析），直观展示为何被判定为隧道。
    *   展示关联的 **主机进程**（通过时间窗口关联算法），直接指出是哪个 PID 发出了这个 DNS 请求。

#### 场景 B：无文件攻击/内存注入溯源
*   **可视化表达**：
    *   在图谱中，某个进程节点呈现“虚线边框”或“幽灵图标”，表示该进程无对应磁盘文件或已被 unlink。
    *   关联的 `ptrace` 或 `memfd_create` 系统调用在详情面板中高亮警告。

#### 场景 C：多阶段攻击演示（靶场验证）
*   **故事线模式 (Storyline View)**：
    *   将离散的日志转化为自然语言的故事线：
        > 10:00:01 - 外部 IP 访问 Web 服务 (Initial Access)
        > 10:00:05 - Apache 进程启动了 /bin/sh (Execution)
        > 10:00:10 - /bin/sh 下载了 unknown 文件 (Defense Evasion)
        > 10:05:00 - 该文件建立了通往 1.2.3.4 的 SSH 隧道 (C2)
    *   每一步都可以点击回溯到原始日志。

### 4. 推荐前端技术栈
*   **框架**: React 或 Vue 3 (组件化开发)。
*   **图可视化**: **Cytoscape.js** (功能最强，适合网络安全拓扑) 或 **AntV G6** (阿里开源，图分析能力强)。
*   **图表库**: **ECharts** (百度开源，极其丰富的图表，适合做仪表盘)。
*   **UI 组件库**: Ant Design Pro 或 Element Plus (暗黑模式支持好)。

这个前端设计方案能够完整覆盖您提出的多源数据融合、攻击链分析、溯源归因及靶场验证需求，将复杂的底层逻辑转化为可视化的安全情报。
